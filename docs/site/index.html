Core features

Security-focused defaults, strong cryptographic design, and operational features that matter when the vault is on the critical path.

Encrypted data at rest

AES-256-GCM with AAD binding prevents record swapping and cross-vault replay attacks.

Member access control

Owner, writer, and reader roles with per-member key wrapping. Invite members via time-limited, passphrase-protected links.

Epoch-based key rotation

Adding or revoking members rotates the vault KEK and re-wraps all item keys atomically in a single transaction.

Rollback detection

Epoch cache detects storage rollback attacks, preventing "undo" of revocations at the storage layer.

Encrypted audit trail

Tamper-evident hash-chained audit entries. Every action is logged and exportable for compliance and forensics.

Pluggable storage

In-memory for tests, BBolt for single-instance deployments, and PostgreSQL for multi-instance HA.

Built-in Certificate Authority

Turn any vault into a CA: issue, renew, and revoke certificates. CRL publishing, CSR signing, and optional PKCS#11 HSM backend.

Encrypted vault export

Export full vault backups encrypted with a passphrase. Portable, offline-readable backup files for disaster recovery.

Authentication & MFA

Layered authentication with phishing-resistant passkeys, time-based one-time passwords, recovery codes, and step-up re-authentication for sensitive operations.

WebAuthn / passkeys

Phishing-resistant FIDO2 authentication. Register multiple passkeys with labels and usage tracking.

TOTP two-factor

Standard 6-digit TOTP codes compatible with any authenticator app. Enable or disable at any time.

Step-up re-authentication

Sensitive operations (export, revoke, role changes, private key reveal) require re-authentication with a 5-minute window.

Passkey policies

Choose between "required" (passkeys mandatory, password login blocked) or "optional" (password + TOTP still allowed).

Recovery codes

Single-use break-glass codes for account recovery. Generated once and stored as SHA-256 hashes — plaintext never persisted.

Vault sharing & invites

Invite members via time-limited links with embedded passphrases (never sent to server). Role assignment at invite time.

How it works

Layered keys with explicit binding

IronHand uses a layered key hierarchy: credentials derive a Master Unlock Key (MUK), which derives a per-vault record key for sealing storage records. A per-epoch KEK wraps per-item DEKs, and fields are encrypted independently.

  • Two-secret MUK derivation — passphrase + secret key, combined via HKDF
  • X25519 member wrapping — KEK distribution uses asymmetric key agreement
  • Per-field encryption — compartmentalised data protection
  • Atomic epoch rotation — real revocation in a single transaction
  • AAD binding — vault ID, item ID, and epoch bound to every ciphertext

Security posture

Designed to be safe by default, with defence in depth across the entire stack.

Hardened defaults

  • Cookie-based sessions with CSRF protection on all mutating endpoints
  • Security headers: CSP, HSTS, X-Frame-Options, Permissions-Policy
  • Rate limiting per-account, per-IP, and global with exponential backoff
  • Proxy headers ignored unless trusted proxy CIDRs are configured
  • Encrypted audit trail with tamper-evident hash chaining
  • Step-up re-authentication for sensitive operations (5-minute TTL)
  • Recovery codes stored as SHA-256 hashes, single-use

Threat model

IronHand protects data at rest and detects rollback/swap attacks in the storage layer. The MUK is derived from two independent secrets (passphrase + secret key) to resist single-factor compromise.

It does not protect against runtime compromise (memory dumps), host side-channels, or storage-layer DoS.

Frequently asked questions

Can I run IronHand as a multi-instance service?

Yes. Use the PostgreSQL backend and configure trusted proxy CIDRs behind your reverse proxy or load balancer. Sessions are stored server-side and shared across instances.

How does member revocation work?

Revocation triggers an epoch rotation: a new KEK is generated, re-wrapped to remaining active members, and all item keys are re-encrypted with new AAD binding — all in one atomic transaction. The revoked member's key material becomes cryptographically useless.

What MFA methods are supported?

IronHand supports WebAuthn/passkeys (FIDO2) and TOTP (time-based one-time passwords compatible with any authenticator app). You can configure passkey enforcement policy ("required" or "optional") and enable or disable TOTP at any time. Sensitive operations require step-up re-authentication via either method.

What is step-up authentication?

Step-up auth is a second layer of verification required for sensitive operations like exporting a vault, revealing private keys, revoking members, changing roles, or creating invites. After verifying via TOTP or passkey, you get a 5-minute window before re-authentication is needed again.

How do vault invites work?

Vault owners and writers can create time-limited invite links (1-hour expiry) with a specific role. The passphrase is embedded in the URL fragment (never sent to the server). The invitee opens the link, and their credentials are wrapped into the vault's key hierarchy.

Can I export and back up my vault?

Yes. The export feature creates an encrypted backup file protected by a passphrase you choose. The backup contains all vault items and can be used for disaster recovery. Exporting requires step-up re-authentication.
